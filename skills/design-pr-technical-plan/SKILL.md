# PR 技术方案设计

## 目标

把上一步的 PR 机会分析转化为可执行的技术方案，并单独整理面试叙事。默认产出两个 Markdown 文档：技术方案设计文档和面试叙事文档；面试叙事文档必须链接到技术方案设计文档。

## 输出落盘规则

- 先确定目标开源项目根目录 `OPEN_SOURCE_ROOT`：优先使用用户提供的本地仓库路径；若当前目录就是目标仓库，则使用当前目录；不要把安装本 skill 的仓库误当成目标仓库。
- 在生成文档前确保 `OPEN_SOURCE_ROOT/tmp/` 存在。
- 默认输出文件必须写入：
  - `OPEN_SOURCE_ROOT/tmp/technical-plan.md`
  - `OPEN_SOURCE_ROOT/tmp/interview-narrative.md`
- 面试叙事文档中的“关联文档”必须链接到实际生成的技术方案文档路径，默认使用 `./technical-plan.md` 或从 `interview-narrative.md` 到 `technical-plan.md` 的相对链接。
- 如果用户只指定文件名，仍写入 `OPEN_SOURCE_ROOT/tmp/`；只有用户明确指定其他目录时，才按用户指定目录执行。
- 聊天回复只给方案摘要、推荐方案、风险、验证重点和已写入文件的绝对路径，不要把完整长文档只放在聊天中。
- `tmp/` 中的方案文档默认是本地工作产物，不要纳入 commit 或 PR，除非用户明确要求。

## 输入要求

从用户提供的分析报告或上下文中提取这些信息：

- 项目地址或本地仓库路径。
- 项目核心架构，例如 ReAct 单 Agent 循环、workflow graph、多 Agent 协作、工具路由层等。
- 选定贡献方向，例如 checkpoint、human-in-the-loop、tool retrieval、MCP 集成、streaming、eval pipeline。
- 涉及的核心文件、函数、类或接口。
- 当前实现的缺陷或缺失。
- 约束条件，例如不破坏公共 API、兼容当前 Python/Node 版本、单个 PR 尽量控制在 300 行以内。

若关键信息缺失，先尝试从本地仓库、`git remote -v`、README、Issue 内容和源码中补齐；仍无法补齐时，明确列出缺失项并给出基于现有证据的保守方案。

## 工作流

### 1. 复核上下文

在开始方案设计前，重新阅读分析报告中列出的核心文件。若用户只给了仓库地址，应先获取或检查代码；若只能访问本地 fork，应基于本地代码分析。

复核时重点确认：

- 缺陷是否真实存在，而不是分析报告中的推测。
- 当前模块边界和公共 API 是什么。
- 现有测试、示例和文档如何覆盖相关行为。
- 是否已有未完成实现、TODO、Issue 讨论或维护者偏好。

输出方案时把关键判断落到具体文件、函数、类或接口，避免只写抽象概念。

### 2. 定义问题边界

先明确本次 PR 要解决的唯一核心问题，再列出不做的事情。边界要足够小，能支撑一个开源维护者愿意 review 的 PR。

必须包含：

- 要解决的核心问题：一句话。
- 不在本次 PR 范围内的问题：2-3 条。
- 成功标准：尽量可验证、可测试、可演示。
- 约束条件：公共 API、版本兼容、改动规模、向后兼容和维护成本。

### 3. 设计 2-3 个方案

每个方案都要能独立成立，不要只是同一个方案的措辞变化。对于每个方案说明：

- 核心思路。
- 新增或修改的核心抽象，例如 class、interface、function、hook、event、schema。
- 数据流：输入、处理、输出。
- 与现有模块的集成点。
- 关键接口签名或伪代码草稿。
- 优点、缺点和适用场景。

如果项目是 Python，接口草稿优先用 Python；如果是 TypeScript/React/Node，优先用 TypeScript。多语言项目选择最贴近改动点的语言。

### 4. 选择推荐方案

用对比矩阵评估 2-3 个方案：

- 实现复杂度。
- 对现有代码侵入性。
- 可扩展性。
- 预计 PR 通过难度。
- 面试叙事价值。

推荐方案应优先满足：最小侵入、最容易验证、维护者最容易接受、能体现清楚的工程取舍。不要为了面试叙事价值选择明显过重的方案。

### 5. 拆解实现规划

基于推荐方案输出文件级改动清单：

- 需要新增的文件：路径、职责、预计行数。
- 需要修改的文件：路径、修改内容、改动幅度。
- 有序实现步骤：每一步都是可独立验证的小目标，完成后项目仍应能运行。
- 验证策略：手动验证、单元测试、边界情况。

实现步骤应避免“实现全部逻辑”这种大块描述，要拆成维护者能理解、开发者能执行的小动作。

### 6. 准备维护者沟通

在写代码前，为用户生成一段英文评论草稿，不超过 150 字。内容包含：

- 客观描述发现的问题。
- 简要说明提议的解决方向。
- 询问维护者是否接受此类 PR，以及是否已有计划或偏好。

语气专业、克制、具体。不要承诺已经完成实现，不要夸大影响。

### 7. 整理面试叙事

在技术方案文档完成后，再把方案转成面试可讲的结构。面试叙事必须引用技术方案文档链接，便于后续复盘时从叙事跳回完整设计依据。

- 背景：项目是什么、解决什么问题、为什么选它。
- 贡献：用户具体引入或改进了什么。
- 过程：3-5 个关键技术决策，不要只列写代码步骤。
- 成果：PR 状态、影响、指标、维护者反馈。
- 困难：1-2 个真实技术卡点，包含排查过程和结论。
- 反思：技术、流程、范围上的改进空间。
- 核心能力：2-3 条，和面试考察维度对应。

## 输出文件

默认生成两个 Markdown 交付物：

- `OPEN_SOURCE_ROOT/tmp/technical-plan.md`：使用 `references/technical-plan-template.md`，包含上下文复核、问题边界、方案设计、方案对比、实现规划、验证策略和维护者沟通草稿。
- `OPEN_SOURCE_ROOT/tmp/interview-narrative.md`：使用 `references/interview-narrative-template.md`，包含面试叙事框架，并在“关联文档”处链接 `technical-plan.md`。

如果用户指定文件名但未指定目录，仍放在 `OPEN_SOURCE_ROOT/tmp/` 下；只有用户明确指定其他目录时，才按用户指定目录执行。若用户只要求其中一部分，只生成对应文档，但仍必须落盘并保留证据、边界和风险。

## 质量标准

- 先复核代码，再设计方案；不要只基于分析报告继续推演。
- 所有关键结论都要能追溯到文件、函数、类、Issue 或维护者评论。
- 优先小 PR、可测试 PR、维护者容易 review 的 PR。
- 明确标注推断、风险和需要维护者确认的点。
- 不建议破坏公共 API、重写核心架构或一次性改太多模块。
- 维护者评论必须英文，其余输出默认中文。
