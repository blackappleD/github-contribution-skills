# 默认 PR 描述模板

当目标仓库没有提供 PR 模板或提交说明时，使用以下模板。若目标仓库已有模板，优先使用目标仓库模板。

## 标题

```text
{type}({scope}): {一句话描述}
```

`type` 优先从以下值中选择：

- `feat`
- `fix`
- `refactor`
- `test`
- `docs`

## PR Body

```markdown
## Problem

{描述当前存在的问题，客观陈述，不带情绪。说明用户路径、代码路径或维护成本上的具体影响。}

## Solution

{描述本 PR 的解决思路，以及为什么选择这个方案。必要时说明替代方案和取舍。}

## Changes

- {新增 / 修改了什么}
- {新增 / 修改了什么}
- {新增 / 修改了什么}

## Testing

- {如何验证这个改动有效}
- {测试用例覆盖了哪些场景}
- {未运行的测试及原因，如无则删除}

## Notes for Reviewer

{需要 reviewer 重点关注的地方，或有意为之的设计决策。没有特别说明时写 “None.”}
```

## 填写要求

- Problem 只描述事实，不写情绪化表达。
- Solution 写清楚为什么这个方案比替代方案更适合当前 PR。
- Changes 使用项目真实文件和行为描述，不堆砌实现细节。
- Testing 只列实际运行过的命令、测试和手动验证。
- Notes for Reviewer 聚焦边界、兼容性、风险和需要确认的设计决策。
