# PR 技术方案实现

## 目标

把已经确认的 PR 技术方案落地为可运行、可测试、便于维护者 review 的代码改动。执行时优先保证现有用法不被破坏、改动范围不越界、每个实现步骤都可验证。

## 输出落盘规则

- 先确定目标开源项目根目录 `OPEN_SOURCE_ROOT`：优先使用技术方案或用户提供的 `LOCAL_PATH`；若当前目录就是目标仓库，则使用当前目录；不要把安装本 skill 的仓库误当成目标仓库。
- 在生成交付报告前确保 `OPEN_SOURCE_ROOT/tmp/` 存在。
- 实现交付报告必须写入 `OPEN_SOURCE_ROOT/tmp/implementation-report.md`，不能只在聊天回复中输出。
- 报告中必须包含改动摘要、接口检查点、验证结果、与方案偏差、剩余风险和 2-4 个 Conventional Commit 提交计划。
- 如果用户只指定文件名，仍写入 `OPEN_SOURCE_ROOT/tmp/`；只有用户明确指定其他目录时，才按用户指定目录执行。
- 聊天回复只给改动摘要、验证结果、剩余风险和已写入文件的绝对路径。
- `tmp/` 中的实现报告默认是本地工作产物，不要纳入 commit 或 PR，除非用户明确要求。

## 输入要求

从用户提供的技术方案或上下文中提取：

- 项目地址：`REPO_URL`
- 本地路径：`LOCAL_PATH`
- 选定方案：方案名称和核心思路。
- 问题边界：要解决的问题和不在范围内的问题。
- 成功标准：判断方案有效的标准。
- 需要新增的文件：文件路径和职责。
- 需要修改的文件：文件路径和修改内容。
- 实现步骤：有序步骤列表。
- 验证策略：手动验证、单元测试、边界情况。

缺少关键信息时，先从技术方案文档、本地仓库和现有测试中补齐；仍无法补齐时先说明缺口，不要猜测后直接大改。

## 实现前准备

写任何代码前，完成只读检查：

1. 阅读所有需要修改的文件，理解当前实现。
2. 检查项目代码风格规范：
   - 配置文件：`.eslintrc`、`eslint.config.*`、`prettier.config.*`、`pyproject.toml`、`ruff.toml`、`mypy.ini`、`tsconfig.json` 等。
   - 命名风格：`snake_case`、`camelCase`、`PascalCase` 等。
   - 注释、docstring、类型标注和错误处理习惯。
   - 测试目录、测试文件命名和断言风格。
3. 查看 `git status --short --branch`，识别用户已有改动，不覆盖、不回滚无关改动。
4. 尽量跑通 README quickstart、现有测试或与改动范围最相关的最小测试集，确认基线状态。

准备完成后再开始编辑。若基线测试失败，记录失败原因；如果失败与本次任务无关，可以继续实现，但最终必须说明剩余风险。

## 实现原则

- 严格按技术方案中的实现步骤推进；若发现实际代码结构要求调整顺序，说明原因。
- 每完成一个小步骤，项目应保持可运行，不留下明显残缺代码。
- 优先扩展现有抽象；必须修改现有函数时，只改必要部分。
- 新增公共接口保持向后兼容。
- 命名、注释、类型标注、错误处理和测试风格必须贴合项目现有代码。
- 不引入新依赖，除非用户明确同意或技术方案已确认该依赖。
- 不执行 `git commit`、`git push`、`git reset --hard`、`git checkout --` 等高影响 Git 操作，除非用户明确要求。

## 实现流程

### Step 1：接口定义

先定义新增的核心接口、类签名、函数签名或类型，不写完整业务逻辑。占位实现可以使用项目惯用方式，例如 `pass`、`raise NotImplementedError`、`todo!()` 或最小可返回值。

完成后输出一个接口检查点：

- 新增了哪些文件、接口、类或函数。
- 这些接口如何衔接现有调用链。
- 是否发现方案设计遗漏的依赖、冲突或边界。

如果用户明确要求分步确认，在这里暂停等待确认。若用户要求一次性完成实现，继续进入 Step 2，但最终报告中必须保留接口检查点摘要。

### Step 2：核心逻辑实现

按照实现步骤依次完成具体逻辑。每个文件完成后记录：

- 文件路径。
- 实现了什么。
- 与方案设计是否有偏差；若有，说明原因。

若伪代码需要适配实际代码结构，可以自行调整；若调整导致改动范围扩大或改变公共 API，必须先停下来说明。

### Step 3：集成到现有代码

把新模块接入现有调用链。重点检查：

- 原有调用方式是否仍然有效。
- 新功能触发路径是否完整。
- 错误处理、默认值、空输入和异常路径是否覆盖。
- 文档、导出入口、类型声明或注册表是否需要同步更新。

### Step 4：验证

按技术方案中的验证策略逐项执行。

手动验证：

- 优先使用项目现有 demo、CLI、example 或 README quickstart。
- 若技术方案要求最小 demo 脚本，只在项目已有约定位置创建；没有约定时，优先用临时命令或现有测试替代，不在项目根目录留下草稿脚本。

单元测试：

- 正常路径：核心功能可按预期工作。
- 边界情况：空输入、超长输入、工具调用失败、外部服务失败或项目相关异常。
- 兼容性：原有用法不受影响。

测试文件放在项目现有测试目录下，命名风格与已有测试一致。

## 必须暂停的情况

遇到以下情况时，先停下来告诉用户，不要自行决定：

- 发现方案设计有遗漏，需要改动超出预定范围的文件。
- 发现与现有代码冲突，无法在不破坏兼容性的前提下实现。
- 有两种实现方式各有明显利弊，需要人工决策。
- 需要引入新的第三方依赖。
- 需要执行数据库迁移、生产操作、远程部署、批量删除或高影响 Git 操作。

## 可自行处理但需说明的情况

遇到以下情况可以继续处理，但最终报告要说明：

- 方案伪代码需要调整以适配实际代码结构。
- 需要补充方案中未提及的错误处理逻辑。
- 测试用例具体写法与方案描述略有不同。
- 基线测试失败但与本次改动无关。

## 交付

实现完成后，读取 `references/implementation-report-template.md`，输出：

- 改动摘要。
- 验证结果。
- 与方案偏差。
- 剩余风险。
- 2-4 个 Conventional Commit 提交计划。

完整交付报告必须写入 `OPEN_SOURCE_ROOT/tmp/implementation-report.md`；聊天回复中只保留摘要、验证结果和报告路径。
