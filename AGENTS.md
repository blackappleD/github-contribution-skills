# AGENTS.md

## 项目说明

- 项目名称：github-contribution-skills
- 项目类型：Codex skill 集合项目
- 主要技术栈：Markdown, YAML, Codex Skills
- 项目根目录：当前 `AGENTS.md` 所在目录

## 语言偏好

- 默认使用中文与用户沟通。
- 代码、变量、函数、文件名遵循项目现有语言和命名习惯。
- 注释优先使用中文，除非项目已有明确英文注释风格。
- 提交信息使用英文 Conventional Commits，必要时可在正文补充中文说明。

## 开发习惯

- 开始任务前先阅读项目结构、README、配置文件、入口文件和相关测试。
- 优先沿用项目已有架构、依赖、代码风格和目录组织。
- 改动保持聚焦，只处理当前任务相关内容，不做无关重构。
- 搜索文件和文本时优先使用 `rg` / `rg --files`。
- 修改前注意当前工作区可能已有用户改动，不覆盖、不回滚、不清理非本次任务产生的内容。
- 新增依赖前先确认项目是否已有等价方案；避免引入过重依赖。
- 涉及配置、构建、部署、权限、数据迁移等高影响改动时，先说明风险和方案。

## Git 规范

- 新项目应先执行 `git init`，并确保项目根目录存在本文件。
- 开始开发前查看 `git status`，结束前再次查看变更范围。
- 不要在未获得明确要求时执行 `git commit`、`git push`、`git reset --hard`、`git checkout --` 等操作。
- 不要回滚用户或其他工具已经产生的改动，除非用户明确要求。
- 分支命名建议：
  - `feat/<short-name>`
  - `fix/<short-name>`
  - `chore/<short-name>`
  - `docs/<short-name>`
- Commit message 使用 Conventional Commits：
  - `feat: add user profile page`
  - `fix: handle empty search results`
  - `chore: update dependencies`
  - `docs: clarify setup steps`

## 验证习惯

- 能运行测试就运行相关测试，优先选择与改动范围匹配的最小测试集。
- 前端改动需要尽量做本地启动和浏览器验证。
- 修复 bug 时，优先补充或更新能覆盖该问题的测试。
- 如果无法运行测试，需要在最终回复中说明原因和剩余风险。

## 文档习惯

- 用户可见行为、启动方式、环境变量、部署流程有变化时，同步更新 README 或相关文档。
- 不写空泛文档，不新增与项目无关的说明文件。
- 临时分析、草稿、脚本不要放在项目根目录，除非项目已有对应约定。
- 新增 skill 时放在 `skills/<skill-name>/`，每个 skill 至少包含 `SKILL.md`。
- skill 名称使用小写字母、数字和连字符，避免空格、下划线和过长名称。

## 安全与配置

- 不提交密钥、令牌、密码、私钥、`.env`、生产数据库连接串等敏感信息。
- 使用 `.env.example` 记录必要环境变量名和说明。
- 远程服务器、SSH、部署凭据只能在当前项目明确授权的场景下使用。
- 生产数据、线上服务、数据库迁移等操作需要格外谨慎，执行前先确认目标和影响范围。

## 任务完成标准

- 代码已实现当前请求。
- 相关测试或验证已完成，或明确说明无法验证的原因。
- 最终回复简洁说明改了什么、验证了什么、还剩什么风险。
